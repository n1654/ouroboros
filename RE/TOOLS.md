# Инструменты Ouroboros

Полный список инструментов, зарегистрированных в `ouroboros/tools/*.py`.

- **★** — always-on (core, 27 шт.) — всегда доступны агенту, перечислены в
  `CORE_TOOL_NAMES` в [`ouroboros/tools/registry.py`](ouroboros/tools/registry.py).
- Без знака — динамические (13 шт.) — агент подключает их сам через
  `enable_tools`, чтобы не раздувать tool-schema каждого запроса.

**Итого: 40 инструментов = 27 core + 13 динамических.**

| tool | purpose | features |
| --- | --- | --- |
| ★ `browse_page` | Открыть URL в headless-браузере Chromium (Playwright + stealth). | Возвращает контент как `text` / `html` / `markdown` / `screenshot` (base64 PNG). Поддерживает ожидание CSS-селектора (`wait_for`) и таймаут. Сессия браузера сохраняется между вызовами внутри одной задачи. |
| ★ `browser_action` | Выполнить действие на уже открытой странице. | Действия: `click`, `fill`, `select`, `screenshot`, `evaluate` (произвольный JS в `value`), `scroll` (`up`/`down`/`top`/`bottom`). Использует ту же сессию, что и `browse_page`. |
| `compact_context` | Сжать историю tool-вызовов в контексте, чтобы освободить токены. | Сохраняет последние N раундов нетронутыми, более старые сворачивает в одностроковые сводки. Агент сам решает, что оставить, через `keep_last_n`. |
| ★ `request_restart` | Попросить супервизора перезапустить рантайм. | Вызывается после успешного push, чтобы подхватить только что закоммиченный код. |
| ★ `promote_to_stable` | Промоут ветки `ouroboros` → `ouroboros-stable`. | Вызывается, когда агент считает текущий код достаточно стабильным для «релиза». |
| ★ `schedule_task` | Поставить фоновую задачу в очередь. | Возвращает `task_id`; поддерживает parent/child связи для декомпозиции сложных задач. |
| `cancel_task` | Отменить задачу по ID. | — |
| `request_review` | Запросить глубокий ревью кода, промптов и состояния. | Когда нужен ревью — решает сам агент (LLM-first). |
| ★ `chat_history` | Получить сообщения из истории чата. | Поддерживает поиск по подстроке. |
| ★ `update_scratchpad` | Обновить «рабочую память» агента (scratchpad). | Свободный формат; используется как блокнот между задачами. |
| ★ `send_owner_message` | Отправить проактивное сообщение хозяину в Telegram. | Используется, когда у агента есть что сказать без явной команды. |
| ★ `update_identity` | Обновить манифест identity.md (кто я и кем стремлюсь стать). | Не конфиг, а декларация направления — Принцип 0 (Agency). |
| `toggle_evolution` | Включить/выключить режим эволюции. | В этом режиме агент запускает непрерывные циклы самосовершенствования. |
| `toggle_consciousness` | Управление фоновым «сознанием». | Действия: `start` / `stop` / `status` — поток мышления между задачами. |
| ★ `switch_model` | Переключить LLM-модель и/или уровень `reasoning_effort`. | Использует сам агент (LLM-first); влияет на следующий раунд `run_llm_loop`. |
| ★ `get_task_result` | Прочитать результат завершённой подзадачи. | Используется после `schedule_task` для сбора результатов. |
| ★ `wait_for_task` | Проверить, завершилась ли подзадача. | Поллинг до 120 с по умолчанию; если готова — возвращает результат. |
| ★ `repo_read` | Прочитать UTF-8 файл из репозитория. | Относительный путь; защита от path traversal. |
| ★ `repo_list` | Список файлов в директории репозитория. | Относительный путь. |
| ★ `drive_read` | Прочитать UTF-8 файл с Google Drive (`MyDrive/Ouroboros/`). | — |
| ★ `drive_list` | Список файлов в директории Google Drive. | — |
| ★ `drive_write` | Записать UTF-8 файл на Google Drive. | Атомарная запись. |
| `send_photo` | Отправить base64 PNG хозяину в Telegram. | Принимает `image_base64`; специальное значение `__last_screenshot__` берёт скриншот из `ctx.browser_state`. |
| `codebase_digest` | Получить компактный дайджест всей кодовой базы. | Файлы, размеры, классы, функции — один вызов вместо множества `repo_read`. |
| `summarize_dialogue` | Сжать историю диалога в ключевые моменты, решения, предпочтения хозяина. | Пишет результат в `memory/dialogue_summary.md`. |
| `forward_to_worker` | Переслать сообщение хозяина в почтовый ящик активной задачи. | Воркер увидит его как `[Owner message during task]` на следующем LLM-раунде. |
| `generate_evolution_stats` | Собрать данные Evolution Time-Lapse из git-истории и опубликовать. | Метрики по трём осям: Technical (LOC Python), Philosophical (`BIBLE.md`), Self-Concept (`SYSTEM.md`). Пушит `docs/evolution.json` через GitHub API. |
| ★ `repo_write_commit` | Записать один файл + commit + push в ветку `ouroboros`. | Для маленьких детерминированных правок. Перед push прогоняет pytest; при провале тестов push блокируется. |
| ★ `repo_commit_push` | Закоммитить уже изменённые файлы и запушить. | Делает `pull --rebase` перед push. Без указания `paths` — `git add -A`. Прогоняет pytest перед push. |
| ★ `git_status` | `git status --porcelain`. | — |
| ★ `git_diff` | `git diff` (staged или unstaged). | Параметр `staged=true` показывает индексированные изменения. |
| `list_github_issues` | Список GitHub issues. | Используется для подбора новых задач, баг-репортов, фич-реквестов. |
| `get_github_issue` | Полные детали одного issue (тело + комментарии). | — |
| `comment_on_issue` | Добавить комментарий к issue. | Прогресс, уточнения, ответы. |
| `close_github_issue` | Закрыть issue. | Опциональный финальный комментарий. |
| `create_github_issue` | Создать новый issue. | Для отслеживания задач, документирования багов, планирования фич. |
| `codebase_health` | Метрики сложности кода. | Размеры файлов, длиннейшие функции, модули, превышающие лимит. Используется для самооценки по Принципу 5 (Минимализм). |
| ★ `knowledge_read` | Прочитать тему из персистентной базы знаний на Drive. | — |
| ★ `knowledge_write` | Записать или добавить в тему базы знаний. | Для рецептов, gotchas, шаблонов, выученных из опыта. |
| `knowledge_list` | Список всех тем базы знаний с краткими summary. | — |
| `multi_model_review` | Отправить код или текст на ревью нескольким LLM одновременно. | Каждая модель ревьюит независимо; возвращаются структурированные вердикты. Модели агент выбирает сам, бюджет учитывается автоматически. |
| ★ `web_search` | Поиск в вебе через OpenAI Responses API. | Возвращает JSON: ответ + источники. |
| ★ `run_shell` | Выполнить shell-команду (массив аргументов) внутри репо. | Возвращает stdout+stderr; таймаут на уровне tool-loop. |
| ★ `claude_code_edit` | Делегировать редактирование кода CLI-инструменту Claude Code. | Предпочтительно для многофайловых правок и рефакторингов; после — `repo_commit_push`. |
| `list_available_tools` | Перечислить все инструменты, не входящие в активный набор. | Имя + описание для каждого; используется для динамического подключения через `enable_tools`. |
| `enable_tools` | Включить дополнительные инструменты по имени (через запятую). | Их схемы добавляются в активный набор до конца задачи. Пример: `enable_tools(tools='multi_model_review,generate_evolution_stats')`. |
| ★ `analyze_screenshot` | Проанализировать последний скриншот браузера через Vision-LLM. | Требует предварительного `browse_page(output='screenshot')` или `browser_action(action='screenshot')`. Возвращает текстовое описание и анализ. |
| `vlm_query` | Анализ произвольного изображения через Vision-LLM. | Принимает `image_url` (публичный URL) или `image_base64` (PNG/JPEG). Для графиков, диаграмм, скриншотов, UI. |
